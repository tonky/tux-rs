# WORKLOG

## 2026-04-07 — Profile save re-applies active profile

### Problem
Editing a profile's keyboard brightness or display brightness and saving it didn't reflect immediately on hardware. Activating the profile did work. Root cause: `UpdateProfile` only persisted to disk without re-applying settings.

### Fix
In `update_profile()`, after saving to the store, check if the updated profile is currently active (for the current power state). If so, nudge `assignments_tx` with `send_modify(|_| {})` to trigger `auto_switch_loop`, which re-reads the profile from the store and applies it to hardware. This reuses the existing apply path (no double-apply race risk).

## 2026-04-07 — Battery stats in TUI Info tab

### Problem
Battery information (cycle count, charge level, health, current draw, voltage, etc.) was available via sysfs but not exposed anywhere in the daemon or TUI.

Additionally, ACPI `cycle_count` reports 0 on Tuxedo hardware because the firmware doesn't properly expose it via ACPI `_BIF`/`_BIX`. The real value lives in EC registers 0x04A6/0x04A7 and is exposed by the tuxedo-uniwill driver as `raw_cycle_count`.

### Changes

**tux-core:**
- Added `BatteryInfoResponse` to `dbus_types.rs` — 14 fields covering capacity, status, cycle count, charge/current/voltage, health %, technology, manufacturer/model. All `serde(default)` for forward compat.
- Added roundtrip test.

**tux-daemon:**
- New `read_battery_info()` in `dbus/system.rs` — reads from first `BAT*` in `/sys/class/power_supply/`, converts µAh/µA/µV to mAh/mA/mV, signs current based on charging status, computes health %.
- Prefers `raw_cycle_count` (tuxedo driver EC direct read) over `cycle_count` (ACPI, often 0 on Tuxedo).
- Falls back to `cycle_count` when `raw_cycle_count` is absent (non-Tuxedo hardware).
- New `GetBatteryInfo` D-Bus method on System interface.
- 4 unit tests: full sysfs mock (with raw_cycle_count), no-battery case, charging sign convention, cycle_count fallback.

**tux-tui:**
- Added `get_battery_info()` to `DaemonClient`.
- Added `BatteryInfo(String)` variant to `DbusUpdate` enum.
- Added `battery: BatteryInfoResponse` field to `InfoState` model.
- Fetched battery info in `dbus_task.rs` alongside other one-time data.
- `update.rs` handler deserializes and stores battery info.
- Info tab renders a color-coded "Battery" section when present:
  - Status (color-coded)
  - Charge: current/full in both mAh and Wh
  - Capacity: current full / design full in mAh and Wh, with health % and degradation indicator
  - Cycle count (from EC via raw_cycle_count)
  - Current draw / charge rate in mA and W
  - Voltage (current + design)
  - Technology, manufacturer, model

### Decisions
- Battery stats are fetched once at startup (alongside other info data). They're relatively static; real-time polling would add unnecessary load.
- Current draw sign convention: negative = discharging, positive = charging (sysfs `current_now` is unsigned; we sign it from `status`).
- Health = charge_full / charge_full_design × 100.
- Wh computed as mAh × design_voltage / 1000 for consistent energy comparison.
- `raw_cycle_count` preferred over `cycle_count` because ACPI firmware on Tuxedo Uniwill hardware reports 0 while the EC stores the real value (e.g., 35 cycles).

## 2026-04-06 — Fix keyboard backlight for white-only sysfs keyboards

### Problem
Keyboard brightness changes from TUI were saved but never applied to the laptop keyboard.
On IBP Gen8, the `tuxedo_keyboard` module + `uniwill_wmi` module create `white:kbd_backlight`
in `/sys/class/leds/` (max_brightness=2, i.e. off/low/high). Our daemon only discovered
`rgb:kbd_backlight` (requiring `multi_intensity` file) and ITE USB HID devices — neither
match this hardware. Device table also incorrectly listed `Rgb1Zone`.

### Root Cause
1. `uniwill_wmi` module was not loaded — without it, `tuxedo_keyboard` can't access EC and
   doesn't create any LED sysfs entry.
2. Device table had `KeyboardType::Rgb1Zone` for IBP Gen8 MK1/MK2 — should be `White`.
3. Daemon sysfs discovery only looked for `rgb:kbd_backlight` entries with `multi_intensity`.

### Changes
- **tux-core/src/device_table.rs**: Changed IBP Gen8 MK1 and MK2 from `Rgb1Zone` to `White`.
- **tux-daemon/src/hid/sysfs_kbd.rs**: Added `SysfsWhiteKeyboard` struct that implements
  `KeyboardLed` — brightness-only, color/mode are no-ops, uses rounded scaling for low
  max_brightness values (e.g. 2).
- **tux-daemon/src/hid/discover.rs**: Extended `discover_sysfs_keyboards_in()` to also match
  `white:kbd_backlight*` entries and create `SysfsWhiteKeyboard` instances.
- **tux-tui/src/update.rs**: When `keyboard_type == "white"`, disable Color and Mode form
  fields (brightness-only for white keyboards).
- Added 10 new tests: 4 for SysfsWhiteKeyboard (open, brightness scaling, color noop, off/on)
  and 2 for discovery (white-only, both RGB+white).

### Testing
- Verified hardware: `echo 1 | sudo tee /sys/class/leds/white:kbd_backlight/brightness` lights up keyboard.
- All 571 workspace tests pass (0 failures).

## 2026-04-06 — Fix TUI inconsistent refresh rate

### Problem
TUI refresh was inconsistent — bursts of many renders per second when events queued up, then 1s gaps. Root cause: the main loop rendered once per event with no batching, and the tick timer (1s, unused) still triggered renders.

### Changes

**tux-tui/src/model.rs**: Added `needs_render: bool` field to `Model`. Set `true` on construction so first frame draws immediately.

**tux-tui/src/update.rs**: Both `handle_key()` and `handle_data()` now set `model.needs_render = true` at entry.

**tux-tui/src/main.rs**: Restructured main event loop:
- Old: `draw → recv (blocks for 1 event) → process → loop`
- New: `recv (blocks) → drain all queued events via try_recv → process all → render once if needed`
- Key/resize events render immediately (interactive responsiveness)
- DBus telemetry events set `needs_render` but only render on next tick
- Tick timer changed from 1s to 500ms (serves as frame clock)
- Extracted `dispatch_command()` helper for cleaner command forwarding

**tux-tui/src/dbus_task.rs**: DBus poll interval reduced from 1s to 500ms for fresher telemetry.

### Result
- Consistent ~2 FPS (500ms) for telemetry dashboard updates
- Instant render on keyboard/resize (no waiting for next tick)
- No wasted renders when no state changed
- Event bursts collapsed into single render via batching

## 2026-04-06 — Keyboard brightness/color not applied from TUI

### Problem
Changing keyboard brightness in the TUI saved the setting but the physical keyboard stayed dark.

### Root Cause
Two bugs in `tux-daemon/src/dbus/settings.rs` `set_keyboard_state()`:
1. **Color never forwarded to hardware** — only `set_brightness()` and `set_mode()` were called, but `set_color()` was never invoked. The ITE per-key row_data buffer stayed at all-zeros (black), so even with brightness set, the LEDs displayed nothing.
2. **Mode name case mismatch** — TUI form options were capitalized ("Static", "Breathe", "Rainbow") but the ITE driver expects lowercase ("static", "breathing", "rainbow"). `set_mode("Static")` silently failed via `let _ = ...`.

### Fix
- Added `parse_hex_color()` helper to convert `#RRGGBB` → `Rgb`
- `set_keyboard_state` now calls `guard.set_color(0, color)` before brightness/flush
- Mode name is lowercased before forwarding: `new_state.mode.to_lowercase()`
- TUI form options changed to lowercase to match driver expectations
- Added regression tests: mock keyboard verifies color, brightness and mode are all forwarded; hex color parsing edge cases

### Files Changed
- `tux-daemon/src/dbus/settings.rs` — added color forwarding, mode lowercasing, parse_hex_color, tests
- `tux-tui/src/model.rs` — keyboard form mode options lowercased

## 2026-04-06 — EC fan drift fix (per-tick PWM re-write)

### Problem
Fan speeds would ramp up after setting a low curve profile. Root cause: the Uniwill EC has its own internal fan control loop that overrides stale PWM values within seconds.

### Investigation
- EC has 16-zone fan tables at `0x0f00`-`0x0f5f` with config bits at `0x07c5`/`0x07c6`
- First attempt: programmed EC's native 16-zone fan table → EC adds its own smoothing/ramping on top, fans still drift up (30 → 38 → 44 over 6 seconds)
- Vendor TCC driver explicitly acknowledges the issue in code comments: "ec does set a different fan speed after one or more hours" and "fan speed does change after writing a value"
- Vendor solution: continuously re-write BOTH zone 0 speed AND direct PWM registers every 1 second, unconditionally for all Uniwill devices

### Solution — per-tick PWM re-write at 1s interval (matching vendor)
Abandoned the write-once EC table approach. Instead:

**Kernel module (`tuxedo_uw_fan.c`):**
- `fan0_pwm_store`/`fan1_pwm_store` already handle dual-write: zone 0 speed + direct PWM
- On first PWM write, `__uw_init_fan_table()` sets up single-zone passthrough (0-115°C) as safety net
- `fan_table` sysfs attribute retained for diagnostics but no longer used by daemon

**Daemon (`tux-daemon`):**
- Removed table mode (`has_fan_table`/`table_dirty`/`tick_fan_table()`) from fan engine
- Always uses `tick_custom_curve()` which re-writes PWM on every tick (even within hysteresis)
- Reduced `idle_poll_ms` from 3000ms to 1000ms (matching vendor's 1s interval)
- The kernel's `fanX_pwm_store` automatically handles both EC table zone 0 update and direct PWM register

### Verification
- Fan values stable: fan0=66-68, fan1=80-82 at 55-56°C, ±2 fluctuation, no upward drift
- Profile switching immediate: Quiet→31-34/45-48, High Performance→106-109/120-123
- All 115 tests pass, kernel module unchanged

## 2026-04-06 — Phase 8.6: TCC Compatibility Shim

### What we did
Implemented a backwards-compatible D-Bus interface (`com.tuxedocomputers.tccd`) that allows the original Angular/Electron TUXEDO Control Center GUI to talk to `tux-daemon` without modification. The shim maps TCC's 58-method flat interface (JSON over D-Bus strings) to tux-rs internals.

**Module**: `tux-daemon/src/dbus/tcc_compat.rs` — single file with TCC JSON schema types, conversion layer, and compat interface implementation.

**TCC JSON schema types**: `TccProfile`, `TccFanData`, `TccSettings`, `TccKeyboardCapabilities`, `TccKeyboardBacklightState`, and nested structs — all matching original TypeScript interfaces with precise `serde(rename)` attributes.

**Feature flag**: `tcc-compat` (default-enabled) with `serde_json` as optional dependency. Entire module gated behind `#[cfg(feature = "tcc-compat")]`.

**D-Bus method naming**: Added `#[zbus(name = "...")]` to 23 methods with non-standard casing (TCC uses `JSON`, `SW`, `NVIDIA`, `CTRL`, `TGP`, `ODM`, `YUV` as acronyms that don't follow standard CamelCase). Critical for compat — without these the original GUI can't find the methods.

**ProfileInterface refactor**: Changed `assignments_tx` from `watch::Sender<ProfileAssignments>` to `Arc<watch::Sender<ProfileAssignments>>` so both ProfileInterface and TccCompatInterface can share write access to profile assignments.

## 2026-04-06 — Fan Control Debugging & Kernel Module Fix

### Context
While building the live regression test (tux-tui/tests/live_regression.rs), fan curve
assertions revealed that setting a fan curve via D-Bus had no effect on actual fan
speed. The fan engine was ticking and computing correct PWM values, but the EC
output (readback from sysfs) didn't change.

### Daemon --debug mode
Added `--debug` / `-d` CLI flag to tux-daemon for verbose logging:
- **tux-daemon/src/main.rs**: Added flag parsing, switched from `LevelFilter` to
  `EnvFilter`-based tracing init. Priority: `RUST_LOG` env > `--debug` flag >
  config `log_level` > default "info".
- **Justfile**: Added `daemon-debug` recipe: builds release, stops systemd service,
  runs `./target/release/tux-daemon --debug`.
- Used `journalctl -u tux-daemon` after temporarily setting `--debug` in the
  systemd service to capture fan engine debug ticks.

### Root cause: kernel module bugs (tuxedo-uw-fan)
Compared our `tux-kmod/tuxedo-uw-fan/tuxedo_uw_fan.c` against the vendor driver
(`vendor/tuxedo-drivers/src/tuxedo_io/tuxedo_io.c`). Found two critical differences:

1. **CFG1 enable ordering**: Our shim enabled the custom fan table (`EC_CUSTOM_FAN_CFG1`
   bit 2 at 0x07c6) BEFORE writing the 16-zone table data. The vendor enables it
   AFTER. This caused the EC to read a partially-populated/stale table and ignore
   subsequent speed writes.

2. **Missing direct PWM write**: The vendor's `uw_set_fan()` writes to BOTH the custom
   fan table register (`EC_CPU_FAN_SPEED` at 0x0f20) AND the direct PWM register
   (`EC_FAN0_PWM` at 0x1804). Our shim only wrote to the table register. The direct
   write gives the EC an immediate "kick" to apply the new speed.

### Fixes applied
- **tuxedo_uw_fan.c `__uw_init_fan_table()`**: Moved CFG1 enable (0x07c6 bit 2) to
  AFTER all 16 zones of table data are written. CFG0 (0x07c5 bit 7) stays before
  data writes (matches vendor).
- **tuxedo_uw_fan.c `fan0_pwm_store()` / `fan1_pwm_store()`**: Added direct PWM write
  (`__ec_write(EC_FAN0_PWM, val)` / `__ec_write(EC_FAN1_PWM, val)`) after the table
  register write, so the EC applies the speed immediately.

### Verification
- Rebuilt kernel module, reloaded via `rmmod`/`insmod`
- Direct sysfs writes now change EC fan output (tested 200, 150, 10 — all reflected)
- Daemon with 60% constant curve: EC fan duty went from 76/90 → 114/128 (target ~120)

## 2026-04-22 — Fan engine config change race condition

### Problem
`config_change_bypasses_hysteresis` test failed consistently (both in full suite and in isolation). The engine timed out waiting for PWM to change from 158 to 204 after a config update.

### Root Cause
Race condition in `FanCurveEngine::run()`. When `tokio::select!` picks the sleep branch over `config_rx.changed()`, the next iteration calls `borrow_and_update()` which silently consumes the change notification. But `last_temp`/`last_pwm` are never reset (that only happens in the `changed()` branch). Since temperature hasn't changed and hysteresis is 10°C, the engine re-uses the old PWM value indefinitely.

### Fix
Added `has_changed()` check at the top of the loop before `borrow_and_update()`. If a config change was missed by select, state is reset so the new curve is applied:
```rust
if self.config_rx.has_changed().unwrap_or(false) {
    last_temp = None;
    last_pwm = None;
}
```

### Files Changed
- `tux-daemon/src/fan_engine.rs` — added `has_changed()` guard at loop top

## Unified tuxedo-uniwill kernel module

### Problem
Vendor `tuxedo_keyboard` + `uniwill_wmi` conflict with our `tuxedo_uw_fan` module — all share the same 6 WMI GUIDs (`ABBC0F6D`–`ABBC0F72`) and fight over EC registers. When our module is loaded, vendor keyboard backlight dies. When vendor modules are loaded, our fan control gets WMI write errors.

### Solution
Merged everything into a single unified `tuxedo_uniwill` module (renamed from `tuxedo_uw_fan`). The module now provides:
- **Fan control** — same as before (EC duty, temps, universal fan table, custom fan curves)
- **Keyboard backlight** — LED class devices: white (3/5 levels), 1-zone RGB (multicolor), via standard `/sys/class/leds/` API
- **Lightbar** — 3 separate LED class devices (R/G/B) + animation toggle
- **Fn lock** — sysfs read/write at `/sys/devices/platform/tuxedo-uniwill/fn_lock`
- **Input device** — sparse keymap for ~15 Fn key events, WMI event handler
- **Charging** — charge_profile, charge_priority (existing), plus ac_auto_boot and usb_powershare (new, DMI-gated)

### Fix: Fan curve lost on profile switch

#### Bug report
User repro: create custom profile → edit fan curve → save → switch to another profile → switch back → custom fan curve gone, replaced with default.

#### Root cause
`set_fan_curve()` in `tux-daemon/src/dbus/fan.rs` only updated the runtime `watch::Sender<FanConfig>` — it never persisted the curve to the active profile on disk. When a profile was re-activated, `ProfileApplier::apply()` read the (unmodified) curve from disk, overwriting the runtime curve.

#### Fix
1. **`ProfileStore::update_fan_settings()`** (`tux-daemon/src/profile_store.rs`): New method that updates fan mode, min_speed_percent, and curve in a custom profile and persists to disk. No-ops silently for builtin profiles and missing IDs.

2. **`FanInterface` extended** (`tux-daemon/src/dbus/fan.rs`): Added `store: Arc<RwLock<ProfileStore>>`, `assignments_rx: watch::Receiver<ProfileAssignments>`, `power_rx: watch::Receiver<PowerState>` fields. New `persist_to_active_profile()` helper determines the active profile ID from current power state + assignments, then calls `store.write().update_fan_settings()`.

3. **`set_fan_curve()` and `set_fan_mode()`**: Both now call `persist_to_active_profile()` after updating the runtime config, so curve and mode changes survive profile switches.

4. **`serve_on_bus()` wiring** (`tux-daemon/src/dbus/mod.rs`): Clones store/assignments_rx/power_rx before `SystemInterface::new()` consumes them, passes to `FanInterface::new()`.

#### Regression test
Added `fan_curve_persists_across_profile_switch` integration test (`tux-daemon/tests/integration.rs`):
- Creates custom profile with 4-point curve
- Activates it, sets custom 3-point curve via D-Bus
- Verifies curve applied at runtime
- Switches to `__quiet__` builtin, switches back to custom
- Verifies all 3 custom curve points restored from disk (not reset to default)

All tests pass (577 tests, 0 failures).
