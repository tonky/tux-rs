# Phase 7.4 Review Findings

## Conformance Review (69%)

Core suspend/resume for fans implemented. WMI hotkeys deferred (per spec). Keyboard code path written but runtime vec is empty.

## Quality Review — Findings & Decisions

### CRITICAL

| # | Finding | Decision | Status | Reasoning |
|---|---------|----------|--------|-----------|
| 1 | Mutex poisoning on panic in lock scope | **IGNORE** | — | Extremely unlikely. Lock scopes contain only clone/assign. No complex operations. |
| 2 | on_resume without on_suspend is safe | **NON-ISSUE** | ✅ | Tested — take() returns None, restoration skipped. |
| 3 | fan_config_tx.send() triggers engine re-apply | **VERIFIED** | ✅ | Watch channel notifies engine immediately via changed() select. |

### IMPORTANT

| # | Finding | Decision | Status | Reasoning |
|---|---------|----------|--------|-----------|
| 4 | Missing 1-2 sec resume delay for hardware init | **IGNORE** | — | on_suspend/on_resume are synchronous (not async). Adding tokio::time::sleep would require making them async and propagating through the D-Bus signal handler. The fan engine polls on its own schedule (100ms) and will naturally wait. Keyboard USB re-enumeration takes its own time. The delay is unnecessary complexity. |
| 5 | Keyboard code path empty at runtime (vec![]) | **IGNORE** | — | Deliberate: keyboards are moved into D-Bus ownership. USB devices re-enumerate on resume. turn_off/turn_on code exists for future use if keyboard sharing is needed. |
| 6 | No D-Bus reconnection retry | **IGNORE** | — | systemd manages D-Bus lifecycle. If logind goes away, bigger problems exist. |
