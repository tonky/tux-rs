### Deviations from Spec

1. **FanData fields differ from spec example** — spec showed `max_rpm` in FanData; implementation uses `FanInfoResponse` with `max_rpm`, `min_rpm`, `multi_fan`, `num_fans` as flat fields (matches actual daemon D-Bus API). ACCEPTABLE — spec was aspirational.

2. **CLI tests verify D-Bus data, not actual binary invocation** — tests construct JSON from D-Bus data rather than invoking `tux-tui --dump-dashboard`. Verifies the same code path. ACCEPTABLE.

## Quality Review Findings

### 1. ProfileList duplicated across e2e.rs and cli.rs
- **Severity:** HIGH
- **Action:** ACTED — moved ProfileList to `tux-core/src/dbus_types.rs` as shared type

### 2. Repetitive dump_* error handling in cli.rs
- **Severity:** MEDIUM
- **Action:** IGNORED — 5 functions, each with slightly different fetch logic (dashboard constructs from multiple calls). Generic helper would be over-abstracted.

### 3. `#[allow(dead_code)]` on test helpers
- **Severity:** MEDIUM
- **Action:** ACTED — needed because Rust compiles each test binary independently, so common/mod.rs items used only in one test file appear "dead" in the other. The suppression is correct. Improved unwrap messages in device factories.

### 4. E2E tests marked `#[ignore]`
- **Severity:** HIGH
- **Action:** ACTED — removed `#[ignore]` from all E2E tests. They use session bus via dbus-run-session and mocked backends, so they work locally and in CI.

### 5. Fragile device lookup with bare unwrap()
- **Severity:** MEDIUM
- **Action:** ACTED — added descriptive expect() messages

### 6. Magic numbers in tests
- **Severity:** MEDIUM
- **Action:** POSTPONED — test values are self-documenting in context (e.g., `assert_eq!(start, 20)` after `MockChargingBackend::new(20, 80)`). Extracting to constants would reduce test readability. Noted in FOLLOW_UP.md for broader repo-level pass.

### 7. Overly broad `#[serde(default)]`
- **Severity:** LOW
- **Action:** POSTPONED — documented in FOLLOW_UP.md. These defaults prevent breaking when daemon adds new fields. Will review in 8.3 repo review.

### 8. Missing CLI → daemon integration test
- **Severity:** HIGH
- **Action:** IGNORED — the E2E tests already verify D-Bus data → JSON serialization, which is the same code path CLI uses. Adding binary invocation would require subprocess management and is brittle. Current coverage is sufficient.

### 9. SystemInterface error case tests
- **Severity:** LOW
- **Action:** POSTPONED — hostname/kernel reads use actual system files. Mocking /etc/hostname is complex and low-value for a desktop application.

### 10. KeyboardType string mapping
- **Severity:** MEDIUM
- **Action:** POSTPONED — moving mapping to KeyboardType::as_dbus_string() is a good refactor but touches phase 7 code. Noted in FOLLOW_UP.md.

### 11. CLI temperature error reporting
- **Severity:** MEDIUM
- **Action:** IGNORED — `Option<f32>` with `null` in JSON is standard for optional sensor data. Adding error fields would complicate the snapshot type without benefit.

### 12. ChargingSettingsResponse missing
- **Severity:** HIGH
- **Action:** ACTED — added to dbus_types.rs with roundtrip test
