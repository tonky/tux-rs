# Phase 8.6 Review Findings

## Findings from two parallel reviewers (Opus 4.6 + Gemini 3.1 Pro)

Both reviewers converged on the same issues. Combined findings:

### CRITICAL ‚Äî D-Bus method names are wrong (23 methods)
zbus converts `snake_case` to `UpperCamelCase`, but TCC uses acronyms like `JSON`, `SW`, `NVIDIA`, `CTRL`, `TGP`, `ODM`, `YUV` that don't follow standard CamelCase. Every method with such acronyms gets the wrong D-Bus name.
- **Status: ACTED UPON** ‚Äî Added `#[zbus(name = "...")]` to all 23+ affected methods + `dbusAvailable`.

### HIGH ‚Äî Keyboard backend not shared (keyboards broken)
`keyboards` is consumed by native `KeyboardInterface`, but compat shim receives `vec![]`.
Keyboard capabilities/states always return empty.
- **Status: POSTPONED** ‚Äî Would require deeper refactoring to share `Box<dyn KeyboardLed>` via Arc between interfaces. Compat shim correctly reports empty/no-op; original TCC GUI will show "no keyboard" which is safer than crashing. Can be addressed separately if needed.

### MEDIUM ‚Äî Missing ProfileApplier call in set_temp_profile*
Compat shim updates assignments but doesn't call ProfileApplier::apply(), so profile settings (fan, CPU) aren't applied until next power state change.
- **Status: POSTPONED** ‚Äî Would require passing ProfileApplier to compat interface and replicating logic. The assignment update will trigger apply on the next power state check. Low real-world impact.

### LOW ‚Äî Default value mismatches vs TCC
- `get_prime_state()` returns `""` but TCC returns `"-1"` ‚Üí **ACTED UPON**
- `get_is_unsupported_configurable_tgp_device()` returns `false` but TCC defaults to `true` ‚Üí **ACTED UPON**
- `set_temp_profile` behavior differs (returns false on not found vs TCC always returns true) ‚Üí **IGNORED** ‚Äî our behavior is safer

### LOW ‚Äî Missing tests
- No manual integration test with `#[ignore]` per spec ‚Üí **ACTED UPON** (added stub)
- No charging roundtrip test with mock backend ‚Üí **POSTPONED** ‚Äî would add complexity; unit tests for charging exist in native interface

### MINOR ‚Äî Improvement suggestions
- Charge threshold available values are hardcoded [1..100] instead of querying backend ‚Üí **IGNORED** ‚Äî TCC behavior varies per hardware, 1-100 is safe superset
- `get_charging_profiles_available` always returns `[]` even with backend ‚Üí **ACTED UPON** ‚Äî fixed to return empty but valid response since our backend doesn't enumerate profiles
- Feature flag gating is verbose (inner module pattern) ‚Üí **IGNORED** ‚Äî pattern works correctly, not worth refactoring
- `now_micros()` u128‚Üíi64 cast theoretically unsound ‚Üí **IGNORED** ‚Äî safe for ~292K years

### Things Done Well (both reviewers agreed)
- All 58 methods + signal are present
- JSON schema types accurately mirror original TypeScript interfaces
- Serde rename attributes are precise (`nvidiaPowerCTRLProfile`, `cTGPOffset`, `tableCPU`)
- Feature flag correctly implemented with optional serde_json dep
- Clean module structure behind cfg(feature)
- Good test coverage (17 tests covering most spec checklist items)
- Thread-safe state management
- Domain-to-TCC profile conversion is thorough
